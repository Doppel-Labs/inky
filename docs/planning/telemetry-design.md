# Inky — Telemetry (design note)

## Why
Inky is self-hosted OSS: when someone `git clone`s and runs it, **you have no idea they exist.**
No install count, no active-team count, no feature-usage signal. Every funnel/revenue number in
both strategy memos is therefore a *guess*. Telemetry is the cheap fix that turns guesses into
signal — and it's the prerequisite for the one experiment that decides floor-vs-ceiling
(does `/ask` get used? does the manager buyer convert?).

## The hard constraint: trust
The audience is **privacy-conscious self-hosters** (the whole product ethos is read-only,
least-privilege, "GitHub activity not surveillance"). Silent phone-home would torch the trust
that *is* the funnel. So telemetry MUST be:
- **Opt-in** (or at absolute minimum loudly opt-*out*, disclosed in the README + first-run log line).
- **Anonymous & aggregate** — a random install UUID + event counts only. **Never** org names, repo
  names, contributor logins/emails, commit/PR content, or LLM keys. (Same discipline as the
  no-individual-names rule for committed artifacts.)
- **Self-host-respecting** — a config flag; trivial to turn off; honored everywhere.
- **Transparent** — document exactly what's sent, in the README and in code. "You can read how we
  refuse to over-collect" is itself a trust asset (the ambitious memo's open-core-as-funnel point).

The hosted tier needs none of this finesse — there you own the infra, so usage data is yours by
default. The hard part this note addresses is the **self-host funnel**, the part you can't see today.

## What to measure (priority order)

**1. Funnel — does anyone run this, and do they stick?**
- `instance_started` — fired once when a worker/CLI boots. Turns GitHub stars (vanity) into
  deployed instances (real).
- `heartbeat` — daily/weekly liveness ping. Distinguishes live deployments from one-time trials.
- (retention falls out of heartbeats over time — instances still pinging at day 30.)

**2. Feature usage — which bets are worth building?**
- `standup_run` — with `{ trigger: 'scheduled' | 'command', windowHours, dryRun }`.
- `ask_run` — **the load-bearing one.** Heavy `/ask` use is the explicit green-light in both memos
  to paywall it and proof the manager wants interrogation, not just a report. (Fires once `/ask` ships.)
- Coarse feature flags in use: `{ statsPanel, roadmap, provider }` — counts only, no values.

**3. Loop 1 — the viral coefficient (the ambitious memo's growth thesis)**
- The 🐙 standup footer carries a **"generated by Inky — host yours"** link with an attribution param.
- `footer_link_clicked` (measured at the landing page, not in the bot) ÷ standups posted = the viral
  coefficient. Without measuring it, "the artifact is the ad" is a story you can't optimize or prove.

## Shape (smallest first version)

Config (non-secret, lives in `inky.config.*`):
```jsonc
{
  "telemetry": {
    "enabled": false,                 // opt-in: off until the operator turns it on
    "endpoint": "https://…/t",        // overridable; default = Inky's ingest (or unset)
    "instanceId": "auto"              // random UUID persisted locally; never derived from org/repo
  }
}
```

Event envelope (the *entire* payload — nothing else ships):
```jsonc
{
  "event": "standup_run",
  "instanceId": "b3f1…",             // anonymous, stable per install
  "version": "1.4.0",                 // Inky version (helps support + upgrade signal)
  "ts": 1733400000,
  "props": { "trigger": "scheduled", "windowHours": 24, "dryRun": false }
}
```

Event list (v1): `instance_started` · `heartbeat` · `standup_run` · `ask_run` (later) ·
`footer_link_clicked` (landing-page side).

## Ingest
Keep it boring. Any of: a hosted analytics product with a server-side event API (PostHog, Plausible
events), or a ~20-line serverless function writing to the Phase-6 Postgres. No new infra needed —
the `runs` table substrate already exists for the hosted side; self-host events can land in the same
ingest. Start with whatever's fastest to stand up; the events above are the contract that matters.

## Explicitly out of scope (v1)
No content, no identities, no per-person data, no LLM prompts/outputs, no GitHub data. If a future
need can't be met with an anonymous count, it doesn't ship without a fresh, disclosed opt-in.

## Definition of done (week one)
`telemetry.enabled` flag honored everywhere · the five events firing · a one-line README disclosure +
a first-run log line · a trivial ingest receiving them · the footer link carrying an attribution param.
That's enough to stop flying blind and to start measuring Loop 1.
